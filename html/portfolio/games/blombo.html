Blombo (2022)

Blombo is a top down shooter game in which you shoot enemies until you reach the end of the level, there is a variety of enemies and levels with some bosses. This is the first game I made and was a way of learning the tools and workflow of game development as well as practicing that which I had learnt from tutorials previous to this.

I solely developed this game over a week long period, all code, art, sfx and music was made by me.

❮ ❯

Concept

The concept for this game was fairly barebones but I wanted to make some sort of 2D shooter game. Being my first game it was more an experiment to add as many features as I could and experience making many different aspects of a game. It ended up with a top down shooter game with the main mechanics being shooting individual or small groups of enemies whilst exploring small and fairly linear levels to find keys and exits.

The game had a backstory about escaping a facility with a buddy robot, this was inspired by the likes of portal's facilities and the common dynamic of main character and support buddy character in games and other media (e.g. dogs in many games or the ghost in destiny 2). The story was not really present in the game due to my lack of experience at the time and loss of interest in the project but did influence some level design such as the prison in the final level.

The game as a whole is meant to be played at a fast pace, the player has high health, fast shooting gun and limited vision which promotes constant movement and shooting.


Mechanics

Gun :

  • The gun is a simple recast system firing at set intervals, it uses a mask system to check what its hitting and what it should pass through.
  • The gun uses an animation, partial system and sound effect to provide player feedback on the shots. The different parts of the player feedback are not synced up meaning the gun can be confusing to the player regarding when it fires.

Enemies :

  • There are a verity of enemies which use a set of modular components, many game objects use the same set of scripts such as health, damage and movement scripts, this allowed for faster development, easier debugging along with the other benefits of composition over inheritance.
  • This modular system allowed for many options with enemies with things like if they damage the player, move or if they drop an item.

Keys & buddy :

  • Basic follow script, keys scale speed depending on player distance to add a 'magnetic' effect.
  • Adds interactable elements the broaden the gameplay, buddy was meant to be linked to story to add a character which the player was to become attached too.

FoV system :

  • Later on a FoV system, similar to the one which was used in Escape the Darkness, was made. This was used to hide enemies out of site to make the game more reactive than strategic.

Mechanics

This game was not technically complex, however being my first game there were many bugs and engine issues which I struggled to diagnose and debug. One of the main difficulties was making a working FoV system, having it being real time and accurate enough for some of the smaller game objects like the bars on the prison cells was difficult requiring multiple iterations.

Another major difficulty was working with my poor code quality at the time, c# and oop were relatively new concepts to me at the time and so keeping my codebase organised was a difficult task. One reason which contributed to me losing interest in the project was the increasing difficulty to add new features due to the messy code. This is something I have worked a lot on since.

In reflection there is nothing major I have to comment on, this was a good first step into game development which taught me alot and highlighted a lot of areas I needed to work on, such as engine knowledge, better programming practices and project management.
